# leaf-ffi

The **Swift / C-ABI frontend binding** for leaf — the native-Apple peer of
[`leaf-wasm`](../leaf-wasm). It wraps `leaf-core`'s frontend-neutral `Doc` (the
byte-offset caret model + the AST→glyph `VisualMap`) behind a C ABI generated by
[UniFFI](https://mozilla.github.io/uniffi-rs/), so an AppKit/SwiftUI app can be a
first-class leaf frontend without any GPUI/wgpu render stack.

```
┌────────────┐   C ABI (UniFFI)   ┌──────────────────┐   Cocoa   ┌───────────┐
│  leaf-core │ ◀───────────────── │     leaf-ffi     │ ◀──────── │  Swift UI │
│  Doc/caret │   DocView / cmds   │  LeafDoc (Mutex) │  DocView  │  renderer │
└────────────┘                    └──────────────────┘           └───────────┘
```

## The contract (same as every leaf frontend)

- **Core owns the model.** Text, caret math, selection, undo, and the
  offset⇄position mapping all live in `leaf-core`. Swift never edits a string.
- **The boundary is style *runs*, not glyphs.** Every call returns a `DocView`:
  rows of `Run`s (maximal same-style spans) plus caret/selection/toolbar state —
  one crossing both edits and repaints. See `src/lib.rs`'s module docs.
- **Core owns the grid; Swift owns the pixels.** Core measures in character
  *columns*; the renderer lets Core Text / `NSLayoutManager` shape each row and
  bridges via `caret_ch` / `click_ch` (**UTF-16 offsets**, exactly what
  `NSAttributedString` and `NSTextView` count in). Never `col × cellWidth`.
- **No filesystem.** Built with `leaf-core`'s `fs` feature off. The host persists
  `source()` its own way (a document wrapper, iCloud, …) and calls `markSaved()`.

## Build

One script produces everything an app links:

```sh
rustup target add aarch64-apple-darwin x86_64-apple-darwin \
  aarch64-apple-ios aarch64-apple-ios-sim x86_64-apple-ios-sim
scripts/build-xcframework.sh          # or --debug
```

It emits (under `generated/`, git-ignored):

- `LeafFFI.xcframework` — the Rust staticlib + C headers for macOS, iOS device,
  and iOS simulator.
- `Sources/LeafFFI/leaf_ffi.swift` — the generated Swift over that ABI.

`Package.swift` exposes two library products from this directory (add it as a
local package, `.package(path: "…/crates/leaf-ffi")`):

- **`LeafFFI`** — the low-level binding: `LeafDoc` + the `DocView`/`Row`/`Run`
  value types (generated under `generated/`, rebuilt by the script).
- **`LeafUI`** — a reusable **AppKit/SwiftUI editor** built on it (`Sources/LeafUI/`,
  committed). Use this unless you're writing your own renderer.

## Native renderer (`LeafUI`)

A ready macOS editor surface. `LeafTextView` (an `NSView`) owns presentation and
input; `LeafEditorModel`/`LeafEditor` wrap it for SwiftUI. It follows the same
contract as every leaf frontend — core owns the model, the view owns the pixels —
and draws each already-wrapped `Row` directly (no `NSTextView`/`NSLayoutManager`
wrapping), placing the caret at `caretCh` and hit-testing clicks with Core Text.

```swift
import SwiftUI
import LeafUI

struct ContentView: View {
    @StateObject private var editor =
        try! LeafEditorModel(source: "# Hello\n\nSome *text*.", format: "markdown")

    var body: some View {
        VStack(spacing: 0) {
            HStack {
                Button("B") { editor.toggleBold() }
                    .fontWeight(editor.isActive("bold") ? .bold : .regular)
                Button("I") { editor.toggleItalic() }
                Button("H1") { editor.setHeading(1) }
                Button("H2") { editor.setHeading(2) }
                Button("• List") { editor.toggleList(ordered: false) }
                Spacer()
                Button(editor.isSource ? "Preview" : "Source") { editor.toggleView() }
                if editor.state.dirty { Circle().frame(width: 6, height: 6) }
            }
            .padding(6)
            LeafEditor(model: editor)   // the editing surface
        }
    }
}
```

`LeafEditorModel` exposes every formatting command (`toggleBold`, `setHeading`,
`toggleList`, `insertLink`, `undo`, …), `source()` / `markSaved()` for
persistence, and a `@Published state` (active marks, heading, dirty, view) for
toolbar binding. Keyboard (typing, arrows/word/line/doc motion with shift-select,
delete/word-delete, ⌘B/I/U/E, ⌘Z/⇧⌘Z, ⌘C/X/V/A), mouse (click, shift-click,
double/triple-click select), and the rich HTML clipboard all work out of the box.
Customize fonts/colours via `EditorTheme`. *Inline IME (marked text) is the one
follow-up* — committed CJK/emoji insert correctly, but the composing overlay isn't
drawn yet (the web got that free from `contenteditable`; AppKit needs it explicit).

## Using `LeafFFI` directly (custom renderer)

```swift
import LeafFFI

let doc = try LeafDoc(source: "# Hello\n\nSome *text*.", format: "markdown")
var frame = doc.view()                 // DocView — paint frame.rows

frame = doc.insert(text: "!")          // every command returns the next frame
frame = doc.toggleBold()
frame = doc.moveRight(extend: true)
frame = doc.clickCh(row: 0, ch: 3, extend: false)   // hit-test from a click

// Render loop sketch:
for row in frame.rows {
    for run in row.runs {
        // map run.role ("h1"…"h6", "code", "link", …) → font/size/weight,
        // run.bold/italic/underline/strike → traits, run.sel → selection bg
    }
}
// place the caret at (frame.caretRow, frame.caretCh)   [UTF-16 offset]
```

### Renderer responsibilities

Core hands you *what* to draw; you own *how*. The map to build on the Swift side:

| `DocView` field | Renderer use |
|---|---|
| `rows[].runs[]` | attributed-string runs; `role`→font, flags→traits, `sel`→bg |
| `rows[].heading` | size the whole row (a heading line is one size) |
| `rows[].code` / `code_lang` | draw a tinted, bordered code panel |
| `rows[].decoration` | drawn but holds no caret — skip in hit-testing |
| `caretRow` / `caretCh` | caret position (`caretCh` is UTF-16) |
| `hasSelection` / `anchorRow` / `anchorCh` | selection range + direction |
| `dirty`, `view`, `heading`, `active` | chrome: modified dot, view toggle, toolbar |

Feed input back through the command methods (`insert`, `newline`, `backspace`,
`move*`, `toggle*`, `clickCh`, `setSelection`, …). `leaf-wasm/web/src/editor.js`
is the reference event-loop for a proportional renderer — the same wiring in JS.

## Why UniFFI (and the one `unsafe`)

`DocView` is a rich nested value returned from every call; UniFFI maps
records/enums/`Vec`/`Option` to native Swift structs, so there's no JSON round
trip. Because a UniFFI object is a thread-safe `Arc`, the `Doc` lives behind a
`Mutex`, and `Inner` carries one `unsafe impl Send` — twig's editor handle is a
`NonNull` (hence `!Send`), but every access is serialized through that mutex and
the handle has no thread-affinity. Drive it from the main thread. See the safety
comment in `src/lib.rs`.
